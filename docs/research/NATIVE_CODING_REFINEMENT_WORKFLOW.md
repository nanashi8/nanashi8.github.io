# 母国語コーディング構想を煮詰めるための推奨手順（検証駆動）

**作成日**: 2026年1月14日  
**ステータス**: 運用ドラフト  
**対象**: 日本語を核とした国産AI推進構想（母国語コーディング×UDD）

---

## 0. 目的

構想を「雰囲気の良いスライド」から「検証して前進できる計画」へ変換する。
ここでのゴールは、以下が揃うこと。

- 何を作るか（スコープ）と、何を作らないか（非目標）が明確
- どの条件を満たせば成功か（受け入れ条件・評価指標）が明確
- 失敗したときに原因分解できる（未知の分類・ログ・停止条件）がある
- 反復の単位が小さい（差分最小・実験単位が小さい）

---

## 1. 構想を「検証可能」にするために不足しがちな概念

以下が欠けると、議論は深まるが前進しにくい。

1. **境界条件（Boundaries）**
   - 対象の開発形態（個人開発/チーム、PRレビュー有無、CI有無）
   - 対象の成果物（コード生成だけか、仕様・テスト・監査ログまでか）

2. **失敗の定義（Stop / No-Go）**
   - 何が起きたら「続けない」のか
   - “改善すれば良くなる”のか、“前提が間違い”なのかを切り分ける条件

3. **比較対象（Baselines）**
   - 英語仕様でやった場合
   - 既存のAIコーディング（一般モデル）
   - 人間のみで実装（時間・品質）

4. **評価の設計（Evaluation Design）**
   - 受け入れ条件を“測れる”形にした指標
   - 日本語特性（主語省略、指示語、敬語、粒度調整、専門語混在）を含む課題設計

5. **ガバナンス（責任分界・監査）**
   - どこで人間が最終承認するか
   - 何を記録すれば説明責任を果たせるか

---

## 2. 推奨プロセス（Gate付き）

### Gate 0：前提固定（1日）

- **スコープ**：どのプロジェクト・どの種類の変更を対象にするか
- **非目標**：この試行でやらないこと
- **運用前提**：レビュー/CI/ブランチ戦略/秘密情報の扱い

**成果物**：
- 日本語仕様テンプレで書かれた「対象タスク定義」1件

このリポジトリ向けの例（Gate 0 成果物）:

- [NATIVE_CODING_PILOT_TASK_DEFINITION_NC_01.md](NATIVE_CODING_PILOT_TASK_DEFINITION_NC_01.md)

### Gate 1：最小パイプライン（MLP）構築（1〜2週間）

MLP = Minimum Lovable Pipeline（最小で“検証の回る”パイプライン）。

- 日本語仕様 → 曖昧さlint → 確認質問
- 確定仕様 → 差分生成（小さく）
- テスト生成（最低：回帰 or 受け入れ条件）
- CI実行 → 失敗原因ログ

**成果物**：
- 変更規模が小さいPRを、仕様・テスト付きで複数（例：10件）

### Gate 2：評価ベンチで比較（2〜4週間）

- 事前に定義した課題セットで、同条件で比較
- 日本語の弱点（曖昧さ）を意図的に含めたケースも評価

**成果物**：
- スコア（定量）+ 失敗原因分類（定性）

### Gate 3：ドメイン適応（必要なら、1〜3ヶ月）

- 行政/医療/教育など、語彙・文書形式・責任分界に最適化

**成果物**：
- ドメイン別テンプレ
- 監査ログ形式（固定）

---

## 3. 実験単位（1反復で必ず出すもの）

1反復あたり、必ず次を残す。

- **意図**：日本語仕様（目的/制約/非目標/受け入れ条件/未確定点）
- **仮定**：確認が取れない部分の明示
- **差分**：小さな変更（レビュー可能なサイズ）
- **テスト**：受け入れ条件に紐づくもの
- **観測**：CI結果、失敗時の原因分類

---

## 4. 停止条件（No-Go）テンプレ

以下のいずれかを満たしたら、改善ではなく前提の見直し（No-Go）を検討する。

- **誤実装が検出できない**：テストやレビューを通過して重大な意図不一致が頻発
- **曖昧さ解消に失敗**：確認質問が増えるだけで仕様が確定しない（または誤った確定をする）
- **回帰が止められない**：小さな変更でも壊れる範囲が広すぎる（差分局所性が悪い）
- **監査不能**：意図→差分→テストの追跡が欠落し、説明責任を果たせない
- **情報管理が破綻**：秘密情報・個人情報・契約情報の扱いで運用不可能
